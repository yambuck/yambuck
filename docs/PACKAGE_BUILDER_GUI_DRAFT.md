# Package Builder GUI Draft (Proposal)

Status: draft for review. This is not normative yet.

This document proposes a GUI-first package authoring surface inside Yambuck to improve package creation usability and help maintainers understand the spec through guided workflows.

## Why GUI-First

- lowers authoring friction for app developers
- reinforces the package spec in-context while building
- reduces reliance on manual manifest editing and memory load
- creates a clear path from "new publisher" to valid `.yambuck` output

CLI tooling can still exist, but this draft treats GUI as day-one primary authoring UX.

## Product Surface Proposal

Add a third top-level tab in the app shell:

- `Installer`
- `Installed Apps`
- `Build Package`

`Build Package` contains two primary entry actions:

1. `Create New Package`
2. `Open Existing Package`

## Core Authoring Modes

### 1) Create New Package

Wizard-oriented flow with progressive validation:

1. app identity (`appId`, `appUuid`, display fields)
2. version + publisher + support metadata
3. interfaces (GUI/CLI)
4. targets and payload mapping
5. assets (icon, screenshots, license)
6. review + build

### 2) Open Existing Package

Import a `.yambuck` file, unpack metadata, then enter edit mode:

- view and edit manifest fields
- inspect payload mapping and entrypoints
- update assets and metadata
- rebuild as a new package artifact (`packageUuid` regenerated by default)

## Information Architecture

Single workspace with three panes:

1. left: step navigation and validation state
2. center: editable form for selected step
3. right: live `manifest.json` preview (read-only by default)

Optional toggle: `Edit raw manifest` (advanced mode).

## Validation Model

Validation should run continuously and mirror package-open/install rules from core validation logic.

- field-level validation while typing
- step-level validation before advancing
- full-package validation before build
- errors in plain language with optional technical detail expansion

Rule source remains the owner spec and core validator implementation.

## Build Output Model

On successful build:

- write `.yambuck` to user-selected destination
- show build report:
  - package path
  - package checksum
  - selected targets
  - warnings (if any)

Future option: one-click smoke test launch in Yambuck installer flow.

## Editing Existing Packages

When opening an existing `.yambuck`:

- preserve original manifest values by default
- highlight fields that are invalid by current rules
- require explicit confirmation when changing identity-critical fields (`appId`, `appUuid`)
- default to new `packageUuid` on rebuild

## Guardrails

- keep default flow non-technical
- hide advanced controls by default
- do not introduce compatibility shims for deprecated formats
- enforce canonical camelCase keys in generated manifests

## Incremental Delivery Plan

Phase 1 (MVP):

- create new package
- import/edit/rebuild existing package
- generate valid manifest and package zip
- support one Linux target (`default` variant)

Phase 2:

- multi-target authoring (x86_64, aarch64, riscv64)
- richer asset tooling and checks
- guided dependency checks authoring (`runtimeDependencies` when finalized)

Phase 3:

- optional template presets (GUI app, CLI app, mixed app)
- optional CLI parity commands backed by same core build engine

## Open Questions

1. Tab label: `Build Package` vs `Create Package`?
2. Should `Open Existing Package` unpack to a temp workspace automatically, or support in-place import only?
3. For first release, should raw manifest editing be included or deferred?
4. Should rebuild always regenerate `packageUuid`, or allow explicit override for testing?
